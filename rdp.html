Introduction =============== Data <=> Electricity Software <=> Chip Design This perspective allows us to adopt ideas from, 1) Electronics Engineering 2) Systems Engineering Currently Electronics have better, 1) Testability 2) Longevity 3) Reliability 4) Maintainability 5) Quality than Software Products, which are perpetually new, improved and broken. RDP is built on Low Level Ideas from Electronics, in hopes of replicating its merits. RDP isa Bread-Board Programming. 1) Tracing 2) Step Debugging 3) Profiling 4) Simulation / Mirroring 5) Diagram Generation come free with RDP. To contrast, High Level Frameworks come with a promise, and just that sometimes, of productivity improvement without any mention of debugging. Pre-90's software was complex, innovative and done in Low Level Languages. Can Low Level Ideas be adapted to todays Application Complexity ? Have we forgotten how the wheel was invented ? 2. Influences ============= In descending order, 1) Motherboard * Pipelines * Buses * Interrupts * Processing * Stores 1.1) Max Msp / Pure Data They are a hallmark for Data Flow Programming. They provide a visual wrapper for digital signal processing. Key Concepts: * "Live" Object * m inlets <process> n outlets RDP API is modelled on the above and FSMs. 2) Hoplon Hoplon is a new web framework for clojurescript based on the idea of spreadsheets. Spreadsheets are the most user friendly interface for Data Flow Programming. Key Concepts: * Cell 0 inlet <identity> 1 outlet * Formula n inlets <script> 1 outlet 3) Entity Systems Key Concepts: * Centralized Store * Batch Processing 4) Gulp / RxJS Key Concepts: * Streams * Generators 5) PowerShell / Pipes / XML Key Concepts: * CmdLet 1 inlet <filter> 2 outlets * Serialization * Validation * RSS Feeds * OSC Messaging 4. Definitions ============== The following words are used from an Engineering perspective. The intent of the words is matched, innocently. Gross mistakes will be corrected. 1) Symbol Symbol has, * a name * an object * attrs This idea is taken from FSMs. "route" is a useful attribute for marking signal routing systems. "Unknown", "NotFound" are common symbols. 1.1) NameSpace A NameSpace has, * elements A NameSpace can, * run queries over its elements * bind * unbind This idea is taken from Lisp. 2) Data A Data has, * props A Property has, * a slot * a sacalar A Slot is a name. A Scalar is a Boolean | String | Number | Array A Data can, * init * serialize This is necessary for a language agnostic design. 2.1) Token A Token isa unique Data. A Token has, * a value One can, * stamp a token start and stop are conventional tokens. Stamping helps in token re-use and tracing. 2.2) Singals and Events A Signal isa Data with, * a route * a payload An Event isa Signal with, * a ts This idea is taken from OSC. 2.3) Glitch A Glitch isa Data with, * a name * a context "Unknown", "NotFound", "Invalid" are common glitches. 3) Entity An Entity isa Data with, * a id * tags * parts * a ts 3.1) Part A Part isa Data with, * a name During Flow, Entities gain or lose Parts. 3.2) Cell A Cell isa Entity with, * observers 4) System A System has, * inlets * outlets * state ( stack ) * registers ( map ) A System can, * react * process * emit * show ( a Graphic Object ) One can, * push * interrupt a System. A System can be a, * Generator * Filter * Stream * StateMachine * Sink * Component Component is a System that just reacts. Stream is a System with, * feedback * sysin * sysout * syserr * debug 5) Board Board provides space for holding Systems and Wires. It can, * connect inlets and outlets of any two Systems with Wires * manage systems * mirror It has, * a store * a bus 5.1) Wire A wire has connection information like, * source * sink * outlet * inlet A wire can, * transmit (emitted data) 5.2) Bus A Bus is a NameSpace which can, * trigger a signal/event to Systems. 5.3) Store A Store can, * manage * query * snapshot * recover for entities of a board. 5.
